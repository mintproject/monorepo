Create `scripts/deploy-hasura.sh` with the following behavior:

1. **Header and argument parsing:**
   - `#!/bin/bash` with `set -euo pipefail`
   - Accept optional positional arg for namespace (default: `mint`)
   - Accept optional `--skip-restart` flag to skip rollout restarts (useful when only migrations changed)
   - Accept optional `--dry-run` flag that prints commands without executing

2. **Step 1 - Rollout restarts** (unless --skip-restart):
   - `kubectl rollout restart deployment/mint-model-catalog -n $NAMESPACE`
   - `kubectl rollout restart deployment/mint-hasura -n $NAMESPACE`
   - Wait for both rollouts to complete: `kubectl rollout status deployment/mint-model-catalog -n $NAMESPACE --timeout=120s` and same for mint-hasura
   - Print status after each step

3. **Step 2 - Find hasura pod:**
   - Use label selector `app=mint-hasura` (matching the pattern from /Users/mosorio/bin/mint-shell-hasura)
   - `kubectl get pod -n $NAMESPACE -l app=mint-hasura -o jsonpath='{.items[0].metadata.name}'`
   - Exit with error if no pod found

4. **Step 3 - Apply migrations and sync metadata:**
   - `kubectl exec -n $NAMESPACE $POD -- bash -c "cd /hasura && hasura migrate apply --skip-update-check && hasura metadata apply --skip-update-check"`
   - Note: use `metadata apply` (not `metadata sync` -- `apply` is the standard hasura CLI command that pushes local metadata to the server; `sync` is only available in the hasura console)
   - The hasura project lives at `/hasura` inside the container (the graphql_engine directory is built into the image)
   - Use `--skip-update-check` to avoid interactive prompts in non-TTY exec

5. **Output formatting:**
   - Use a `step()` helper function that prints `==> Step N: description` with the step number
   - Print a final success summary with namespace used

6. **Error handling:**
   - `set -euo pipefail` at top ensures any failure stops the script
   - Trap ERR to print which step failed
   - Check `kubectl` is available at script start
   - Check the cluster is reachable with `kubectl cluster-info --request-timeout=5s` before proceeding

Do NOT use `hasura metadata sync` -- it is a console-only command. Use `hasura metadata apply` which pushes the local metadata directory to the hasura instance.
