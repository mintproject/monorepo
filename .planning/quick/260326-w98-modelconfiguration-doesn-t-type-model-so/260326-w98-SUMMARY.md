# Quick Task 260326-w98: Scope ModelConfiguration type assignment Summary

**One-liner:** Conditional `input['type']` assignment in `service.ts create()` restricted to `modelcatalog_software` table resources, preventing column-not-found errors on POST /modelconfigurations and other non-software-table resources.

## What Was Done

The previous quick task (260326-vn8) added `input['type'] = resourceConfig.typeUri` unconditionally in the `create()` method of `service.ts`. This broke POST requests to resources whose backing Hasura tables lack a `type` column -- most notably `modelcatalog_model_configuration` (used by `/modelconfigurations`) and `modelcatalog_software_version` (used by `/softwareversions`).

The fix wraps the assignment in a conditional:

```typescript
if (resourceConfig.hasuraTable === 'modelcatalog_software') {
  input['type'] = resourceConfig.typeUri
}
```

Only `modelcatalog_software` has a `type` column. All Model subtypes (`models`, `empiricalmodels`, `hybridmodels`, `emulators`, `theory-guidedmodels`, `coupledmodels`, `softwares`) use this table and correctly get their canonical type URI assigned. All other resources do not receive a `type` field in the Hasura INSERT input.

## Decisions Made

- Used `hasuraTable === 'modelcatalog_software'` as the gate rather than adding a new boolean field to `ResourceConfig`. This approach requires no schema changes and directly encodes the database reality: exactly one table has the `type` column.
