<objective>
Fix the blocker: remove broken `user_id` filtering from service.ts and custom-handlers.ts.

Purpose: The `user_id` column does not exist in any `modelcatalog_*` table. The PostgreSQL schema uses `author_id` (FK to `modelcatalog_person`) for author attribution. The v1.8.0 API's `?username=` parameter selected a Fuseki named graph -- there is no equivalent column in PostgreSQL. Any request with `?username=` currently causes a Hasura runtime error (unknown column `user_id`). This is a blocker for all list endpoints.

The fix: Remove all `user_id: { _eq: $username }` WHERE clause injections. The `?username=` query parameter will be accepted but ignored (no-op) since there is no per-user data segregation in the PostgreSQL schema. This matches the real-world behavior: all data was loaded from a single TriG dump and is not partitioned by user.

Output: Updated service.ts and custom-handlers.ts with user_id filtering removed.
</objective>
