**response.ts** -- transforms Hasura query results to v1.8.0 response format:

`transformRow(row: Record<string, any>, resourceConfig: ResourceConfig): Record<string, any>`
Rules (derived from live API inspection documented in 02-RESEARCH.md):
1. `id` field stays as-is (URI string, NOT array-wrapped)
2. `type` field is synthesized from `resourceConfig.typeArray` (e.g., `["Model", "SoftwareDescription"]`)
3. All other scalar fields are wrapped in `[value]` arrays
4. Null/undefined fields are OMITTED entirely (not null, not empty array)
5. Field names are converted from snake_case (Hasura) to camelCase (API)
6. Related objects (from Hasura object_relationships) become nested objects with their own `id` and `type`, recursively transformed
7. Related arrays (from Hasura array_relationships through junction tables) become arrays of nested objects

`transformList(rows: Record<string, any>[], resourceConfig: ResourceConfig): Record<string, any>[]`
Maps transformRow over the array.

**request.ts** -- transforms v1.8.0 request bodies to Hasura insert/update format:

`toHasuraInput(body: Record<string, any>, resourceConfig: ResourceConfig): Record<string, any>`
Rules (reverse of response transform):
1. Unwrap single-element arrays back to scalars
2. Convert camelCase field names to snake_case
3. Extract `id` as-is
4. Ignore `type` field (not stored)
5. Handle nested related objects by extracting their IDs for FK columns

**response.test.ts** -- vitest tests:
1. Test scalar wrapping: `{ id: "https://...", label: "CYCLES" }` -> `{ id: "https://...", type: ["Software"], label: ["CYCLES"] }`
2. Test null omission: `{ id: "https://...", label: "CYCLES", description: null }` -> no `description` field
3. Test snake_case to camelCase: `has_documentation` -> `hasDocumentation`
4. Test nested object: row with `software_version: { id: "...", label: "v1" }` -> nested object with id and type
5. Test request body unwrapping: `{ label: ["CYCLES"] }` -> `{ label: "CYCLES" }`
