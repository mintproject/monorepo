<objective>
Make 23 resource types with null hasuraTable return empty read responses instead of HTTP 501.

Purpose: The verification found that 23 of 46 API resource types return 501 for all CRUD operations because they have `hasuraTable: null` in the registry. These types (catalogidentifiers, constraints, datatransformations, etc.) have no dedicated PostgreSQL tables because the Phase 1 schema did not include them -- they were either value types inlined in the ontology, had zero instances in the TriG data, or were subtypes not separately stored.

The v1.8.0 API returns empty results (`[]`) for queries against resource types that have no data in the named graph. To match this contract, the v2.0.0 API should do the same: return `[]` for list requests and `404` for get-by-id requests on null-table types. Write operations (POST, PUT, DELETE) should continue returning 501 since there is genuinely no backing store.

This closes the "all 46 resource types" gap in the verification while being honest about the data situation -- these types have no data to return, and the existing v1.8.0 API would also return empty results for them.

Output: Updated service.ts with modified read handlers.
</objective>
