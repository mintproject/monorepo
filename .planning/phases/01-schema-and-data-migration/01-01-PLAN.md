IMPORTANT: The TriG data shows that "Software" entities are typed as `sdm#Model` (NOT `sd#Software`). The `software_id` FK in `modelcatalog_software_version` is derived from the INVERSE of the `sd#hasVersion` predicate on the sdm#Model entity. Similarly, `software_version_id` in configuration comes from inverting `sd#hasConfiguration`, and `model_configuration_id` in setup comes from inverting `sd#hasSetup`. The ETL script (Plan 02) will handle this inversion; the schema just needs the FK columns.

NOTE on nullable FKs: `software_id` on software_version should be nullable (NOT NULL would be ideal but some SoftwareVersion entities may not have an identifiable parent sdm#Model). Same for `software_version_id` on configuration and `model_configuration_id` on setup. Make these FK columns nullable with a comment explaining why. The ETL script will log orphans.
