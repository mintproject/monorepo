**etl/extract.py:**
Use `rdflib.Dataset` (NOT Graph -- TriG has named graphs/quads).

Implement extraction functions that return lists of dicts:

1. `extract_software(ds)` -- Extract entities typed as `sdm#Model` (URI `https://w3id.org/okn/o/sdm#Model`). These are the top-level "Software" entities. Expected: ~186 entities.
   Fields: id (URI), label (rdfs:label), description (sd:description), keywords (sd:keywords), license (sd:license), website, date_created, date_published, has_documentation, has_download_url, has_purpose
   Also extract: `sd#hasVersion` links -> list of SoftwareVersion URIs per Software

2. `extract_software_versions(ds)` -- Extract entities typed as `sd#SoftwareVersion`. Expected: 240.
   Fields: id, label, description, keywords, has_usage_notes, date_created, has_source_code, version_id (sd:hasVersionId)
   Also extract: `sd#hasConfiguration` links -> list of ModelConfiguration URIs per SoftwareVersion

3. `extract_model_configurations(ds)` -- Extract entities typed as `sdm#ModelConfiguration` (use the full URI `https://w3id.org/okn/o/sdm#ModelConfiguration`). Expected: 308.
   IMPORTANT: The TriG also has `sd#SoftwareConfiguration` as a secondary type on the same entities -- use sdm#ModelConfiguration as the canonical type.
   Fields: id, label, description, keywords, usage_notes (sd:hasUsageNotes), has_component_location (sd:hasComponentLocation), has_implementation_script_location, has_software_image
   Also extract: `sd#hasSetup` links, `sd#hasInput` links, `sd#hasOutput` links, `sd#hasParameter` links

4. `extract_model_configuration_setups(ds)` -- Extract entities typed as `sdm#ModelConfigurationSetup`. Expected: 757.
   Fields: id, label, description, has_component_location, has_implementation_script_location, has_software_image, has_region
   Also extract: `sd#hasInput`, `sd#hasOutput`, `sd#hasParameter` links

5. `extract_dataset_specifications(ds)` -- Extract entities typed as `sd#DatasetSpecification`. Expected: 5,434.
   Fields: id, label, description, has_format (sd:hasFormat), has_dimensionality (sd:hasDimensionality), position (sd:position)

6. `extract_parameters(ds)` -- Extract entities typed as `sd#Parameter`. Expected: 8,273.
   Fields: id, label, description, has_data_type, has_default_value, has_minimum_accepted_value, has_maximum_accepted_value, has_fixed_value, position
   Also determine parameter_type: if entity ALSO has type `https://w3id.org/wings/export/MINT#Adjustment`, set parameter_type="adjustment", else "standard"

For SPARQL queries: use `OPTIONAL` for all fields except id and label. Use `DISTINCT` in SELECT to handle duplicate results from named graphs. String values should use `str()` to avoid rdflib Literal objects in output.

Print entity counts after each extraction for progress tracking.

**etl/transform.py:**
Implement `transform_all(extracted_data)` that takes the raw extraction output and produces load-ready data structures.

Key transformations:
1. **FK inversion for software_version.software_id:** The TriG has `Software --sd:hasVersion--> SoftwareVersion`. Build a reverse lookup: for each Software entity's hasVersion list, set software_version.software_id = that Software's id. Log any SoftwareVersion that has no parent Software.

2. **FK inversion for configuration.software_version_id:** The TriG has `SoftwareVersion --sd:hasConfiguration--> ModelConfiguration`. Build reverse lookup similarly. Log orphans.

3. **FK inversion for setup.model_configuration_id:** The TriG has `ModelConfiguration --sd:hasSetup--> ModelConfigurationSetup`. Build reverse lookup similarly. Log orphans.

4. **Junction table rows:** From the hasInput/hasOutput/hasParameter links on configurations and setups, produce rows for the 6 junction tables:
   - modelcatalog_configuration_input: (configuration_id, input_id)
   - modelcatalog_configuration_output: (configuration_id, output_id)
   - modelcatalog_configuration_parameter: (configuration_id, parameter_id)
   - modelcatalog_setup_input: (setup_id, input_id)
   - modelcatalog_setup_output: (setup_id, output_id)
   - modelcatalog_setup_parameter: (setup_id, parameter_id)

5. **Deduplication:** Some SPARQL queries may return duplicate entities from multiple named graphs. Deduplicate by id (first occurrence wins).

Return a dict keyed by table name, each containing a list of row-dicts ready for insertion.

**etl/load.py:**
Implement `load_all(transformed_data, conn)` that takes the transform output and a psycopg2 connection.

Loading order MATTERS due to FK constraints:
1. modelcatalog_software (no FK dependencies)
2. modelcatalog_dataset_specification (no FK dependencies)
3. modelcatalog_parameter (no FK dependencies)
4. modelcatalog_software_version (FK to software)
5. modelcatalog_model_configuration (FK to software_version)
6. modelcatalog_model_configuration_setup (FK to model_configuration)
7. Junction tables: configuration_input, configuration_output, configuration_parameter
8. Junction tables: setup_input, setup_output, setup_parameter

For each table:
- Use `psycopg2.extras.execute_batch()` with `page_size=500`
- Use `ON CONFLICT (id) DO NOTHING` for entity tables (to handle reruns safely)
- Use `ON CONFLICT DO NOTHING` for junction tables
- Print row count after each table load
- Wrap entire load in a transaction (single commit at end)

Handle nullable FKs: If a SoftwareVersion has no software_id (orphan), still insert it with software_id=NULL. Log a warning.

Implement `clear_all(conn)` that DELETEs from all tables in reverse order (for reruns/testing). Use TRUNCATE with CASCADE for efficiency.

**etl/validate.py:**
Implement `validate(trig_path, conn)` that:

1. **Count validation:** For each entity type, compare:
   - Source count: SPARQL `SELECT (COUNT(DISTINCT ?uri) as ?count) WHERE { ?uri a <TYPE> }` on the TriG Dataset
   - Target count: `SELECT COUNT(*) FROM modelcatalog_<table>`
   - Print comparison table and PASS/FAIL for each
   - Entity types to validate: Software (sdm#Model), SoftwareVersion, ModelConfiguration, ModelConfigurationSetup, DatasetSpecification, Parameter

2. **Junction table validation:** For each junction table, count rows and report. No source comparison needed (derived data), but count should be > 0.

3. **Sample spot-check:** For 3 well-known entities (pick from TriG: e.g., topoflow_3.5, topoflow_cfg_simple, topoflow_cfg_simple_Baro):
   - Query the entity from PostgreSQL
   - Verify label matches what's in TriG
   - Verify FK relationship exists (e.g., topoflow_cfg_simple's software_version_id points to topoflow_3.5)
   - Verify junction table links exist (e.g., topoflow_cfg_simple has inputs in configuration_input)

4. **Orphan report:** Count entities with NULL FK values (software_version with no software_id, etc.) and report.

5. Return overall PASS/FAIL. FAIL if any entity count mismatch > 0.

**etl/run.py:**
Orchestrator script that:

1. Parses CLI args: `--trig-path` (default from config), `--db-*` (default from env/config), `--clear` (flag to truncate before load), `--validate-only` (skip extract/transform/load)
2. Runs the pipeline: extract -> transform -> load -> validate
3. Prints timing for each phase
4. Exits with code 0 on success, 1 on validation failure

Usage: `python3 etl/run.py --trig-path ../model-catalog-endpoint/data/model-catalog.trig`

Make run.py executable with `if __name__ == "__main__":` block.

IMPORTANT: The entire ETL should be idempotent -- running it twice produces the same result (via ON CONFLICT DO NOTHING or --clear flag).
